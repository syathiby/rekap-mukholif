💡 CATATAN ARSITEKTUR & PENJELASAN LOGIKA KRUSIAL UNTUK AI/DEVELOPER
=============================================================================

1. Konsep "Tutup Buku" (Snapshot Arsip Historis) - `pengaturan/reset-poin/process.php`
   Aplikasi ini mengisolasi data periode tahun ajaran aktif dengan data historis masa lalu.
   Ketika fitur "Tutup Buku" (Reset Poin) dijalankan:
   - Data Rapot, Pelanggaran Kebersihan, dan Total Poin Santri difoto/disalin utuh
     menggunakan klausa `INSERT INTO ... SELECT ...` ke dalam tabel `arsip_data_*`.
   - Proses ini sangat bergantung pada struktur *Denormalisasi*. Artinya, meskipun ID Santri
     disimpan, nama dan atribut teks santri (nama, kelas, kamar, nama musyrif) akan ikut di-copy
     secara string murni (`santri_nama`, `musyrif_nama`). Ini memastikan data rapot 5 tahun 
     lalu tetap bisa dibaca namanya sekalipun akun santri tersebut dihapus hari ini.
   - Setelah sukses di-snapshot, data di tabel operasional akan dihapus bersih (TRUNCATE/DELETE).

2. Aturan Navigasi & File Inclusion (Absolute Pathing)
   Seluruh sistem web ini menggunakan pola pemanggilan `require_once __DIR__ . '/.../init.php'`.
   Ini merupakan pola wajib agar aplikasi tetap kokoh dijalankan di lingkungan manapun (xampp, laragon, server linux) tanpa error _file not found_. `init.php` akan memuat sesi, koneksi DB, dan `helpers.php` yang berisi utilitas pengaman seperti *guard* hak akses.

3. Logika Visibilitas Permanen Pelanggaran "Sangat Berat"
   Modul rekap (seperti `detail_per_santri.php` & `detail_pelanggaran.php`) dibatasi oleh rentang 
   `tanggal mulai` dan `tanggal selesai` dari periode aktif. Namun, ada pengecualian *hard-coded*:
   `AND (tanggal BETWEEN x AND y OR jp.kategori = 'Sangat Berat')`
   Hal ini memastikan pelanggaran dengan status ekstrem/Sangat Berat akan MENEMBUS filter periode 
   dan terus menghantui/menempel di rapor santri tersebut secara permanen. Tetapi pengecualian ini 
   tidak menembus filter *Dropdown Kategori* (jika user memilih untuk memfilter secara eksplisit).

4. Manajemen Izin Berbasis Modular (Role Based Access Control)
   Sistem diamankan menggunakan `guard('nama_izin')` pada `init.php`. 
   Setiap fungsi sensitif (seperti Create, Delete) wajib memanggil `guard()` di baris ke-2 file.
   - Hubungan antara entitas `users` <-> `permissions` dijembatani lewat relasi Many-to-Many 
     di tabel `role_permissions` (Grup Peran) dan `user_permissions` (Izin Khusus Individu).
   - Kedua tabel ini dilengkapi dengan **Foreign Key ON DELETE CASCADE** ke `permissions.id`, 
     sehingga menghapus izin tunggal akan membersihkan semua hak akses terkait secara rekursif di seluruh sistem.

5. Paradigma CRUD dan REST/API
   Aplikasi ini adalah monolith PHP native konvensional yang digabungkan dengan teknik modern (AJAX).
   - File dengan embel `process.php` murni berupa *Action Endpoint* (menerima POST dan melakukan Redirect).
   - Folder berlabel `api/` atau berkas bernama `search_santri.php` adalah *JSON Endpoint* yang 
     menyuplai data untuk *Select2/Autocomplete/Fetch API* di antarmuka frontend (tanpa reload).
